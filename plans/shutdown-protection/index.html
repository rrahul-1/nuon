Runner Process Shutdown Protection

Plan review document — click any section to leave inline comments

Context

When a runner process shutdown is requested (e.g., at 8-hour rotation), two things go wrong:

  1. In-flight jobs get killed — the jobLoop context is cancelled, terminating the running job
  2. New jobs queued during shutdown get stuck — the processjob signal marks jobs "available" for a draining runner that will never pick them up, leading to timeout

Two changes needed:

  1. Runner-side drain: Stop accepting new jobs, finish the in-flight job, then shut down
  2. API-side hold: The processjob signal should not dispatch a job to a runner with a pending shutdown — wait until shutdown completes and a new healthy process appears

Part 1: Runner-Side Drain

Root Cause

The jobLoop uses a single ctx/ctxCancel for both polling and job execution. Stop() cancels everything simultaneously. The ShutdownPoller and jobLoop are decoupled FX components — the only coordination path is fx.Shutdowner.Shutdown() which triggers all OnStop hooks including the job loop's Stop().

Design: Drainer + Two-Context Separation

New: bins/runner/internal/pkg/jobloop/drain.go

An FX singleton that the ShutdownPoller signals and all jobLoop instances listen to.

type Drainer struct {
    drainOnce sync.Once
    drainCh   chan struct{}   // closed when Drain() is called
    mu        sync.Mutex
    doneChs   []<-chan struct{} // one per job loop
}

Methods:

  • NewDrainer() *Drainer — FX constructor
  • Register(ch <-chan struct{}) — each job loop registers its done channel
  • Drain() — closes drainCh, signaling all loops to stop
  • DrainCh() <-chan struct{} — for loops to select on
  • Wait(timeout) error — blocks until all done channels close or timeout

Two-Context Split in jobLoop

Replace the single ctx/ctxCancel with:

  • pollCtx / pollCancel — controls the polling loop; cancelled on drain
  • jobCtx / jobCancel — controls in-flight job execution; only cancelled on hard stop

Part 2: API-Side — Hold Jobs During Runner Shutdown

Root Cause

In processjob signal's startJobExecution(), the runner health-wait loop (lines 251-313) only checks if the runner status is active. It doesn't check whether the runner's process has a pending shutdown. So it marks the job "available" and the draining runner ignores it, leading to AvailableTimeout expiry.

Design: Shutdown-Aware Health Loop

The startJobExecution() health loop already polls runner status. We add a check: does this runner have an open (requested/in-progress) process shutdown? If yes, keep waiting — don't mark the job available yet.

The loop becomes: "wait until runner is active AND no pending shutdown exists."

New Activity: HasOpenProcessShutdown

File: services/ctl-api/.../activities/has_open_process_shutdown.go

type HasOpenProcessShutdownRequest struct {
    RunnerID string `validate:"required"`
}

type HasOpenProcessShutdownResponse struct {
    HasOpenShutdown bool
}

// @temporal-gen-v2 activity
// @by-field RunnerID
func (a *Activities) HasOpenProcessShutdown(
    ctx context.Context,
    req HasOpenProcessShutdownRequest,
) (*HasOpenProcessShutdownResponse, error) {
    var count int64
    res := a.db.WithContext(ctx).
        Model(&app.RunnerProcessShutdown{}).
        Joins("JOIN runner_processes ON ...").
        Where("composite_status->>'status' IN ?",
            []string{"requested", "in-progress"}).
        Count(&count)
    return &HasOpenProcessShutdownResponse{
        HasOpenShutdown: count > 0,
    }, res.Error
}

Follows the exact pattern from hasOpenProcessShutdown() in report_runner_process_terminating.go, but as a Temporal activity querying by runner ID.

Signal Change: processjob/signal.go

In startJobExecution(), the runner-health polling loop changes from:

for attempt := 0; ; attempt++ {
    runnerStatus = AwaitGetRunnerStatusByID(...)
    if runnerStatus == active { break }
    // ... timeout checks ...
    workflow.Sleep(ctx, pollPeriod(attempt))
}

To:

for attempt := 0; ; attempt++ {
    runnerStatus = AwaitGetRunnerStatusByID(...)
    shutdownResp = AwaitHasOpenProcessShutdown(...)

    if runnerStatus == active && !shutdownResp.HasOpenShutdown {
        break  // healthy AND no pending shutdown
    }

    // ... existing timeout checks ...
    workflow.Sleep(ctx, pollPeriod(attempt))
}

The job stays in this loop until the old process drains, shuts down, a new process starts and becomes active, and no open shutdown records remain. Then the job is marked "available" and the new process picks it up.

Safety nets preserved: The existing AvailableTimeout and OverallTimeout still apply. If a new process never comes up, the job eventually times out.

Edge Cases

No in-flight job at drain

pollCtx cancellation interrupts fetch/sleep. Worker exits immediately, Wait() returns instantly.

Long-poll blocking

fetchAvailableJobs derives timeout from pollCtx. Cancellation aborts the HTTP request.

mng process

Gets a Drainer injected — harmless if no job is running. Management jobs are short-lived.

Drain timeout (30m)

Shutdown proceeds anyway. fx.Shutdown triggers hard stop via jobCancel().

os.Exit(155) in runWorker

Must not fire during drain — check drainer state before calling exit.

Job queued during shutdown

processjob signal holds in health loop until new process is active with no open shutdown.

Runner never restarts

OverallTimeout eventually marks the job as timed out.

Multiple shutdowns

HasOpenProcessShutdown counts any open shutdown, so stacked requests are handled.
