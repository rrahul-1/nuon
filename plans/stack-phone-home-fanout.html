Key Design Points

  • Always enqueue — no more existingRunCount > 1 guard
  • Callback replaces polling — await blocks on signal, not 15s poll loop
  • Stale version support — old stack applied? Run copied to newer version automatically
  • Diff always runs — even during provision (first run emits all inputs/roles as "added")
  • updateinstallstackoutputs step removed from provision/reprovision workflows

Why This Happens

A user might have the CloudFormation URL from a previous provision. If a reprovision generates a new version (v2), the user may still apply the old template (v1). The outputs are the same infrastructure, so we propagate the run data to the newer version.

What Gets Propagated

  • A new InstallStackVersionRun is created on v2 with the same HSTORE output data from the v1 run
  • v2 status is set to active with metadata tracking provenance
  • v2's callback is sent, unblocking the provision/reprovision workflow
  • Shared InstallStackOutputs are already updated (keyed by install stack, not version)

Status Metadata

The newer version's status includes provenance tracking:

{
  "status": "active",
  "metadata": {
    "applied_from_version_id": "isv_abc...",
    "applied_from_run_id": "isvr_xyz...",
    "propagated_run_id": "isvr_new..."
  }
}

How the Callback Works

  1. awaitinstallstackversionrun creates a callback.Ref with the current workflow ID and a deterministic signal name: signal-complete-{version_id}
  2. Writes the ref to InstallStackVersion.CallbackRef (new JSONB column)
  3. Calls callback.AwaitWithTimeout(ctx, cb, 180 days) — opens a Temporal signal channel and blocks
  4. When stack-run detects first run, it reads the CallbackRef from the version row and sends success
  5. If a stale version phones home, stack-run reads the newer version's CallbackRef and sends success
  6. The await signal receives the signal, unblocks, and returns

CallbackRef Storage

New JSONB field on InstallStackVersion:

{
  "workflow_id": "queue-install-signals-abc123",
  "signal_name": "signal-complete-isv_xyz789",
  "namespace": "components"
}

Timeout Handling

If no phone home arrives within 180 days:

  • Version status → Expired
  • Workflow step status → Error
  • Error: "Stack was not applied within 180 days"

Callback Sources

A version's callback can be triggered from two paths:

TriggerWhen
Direct first runPhone home matches this version's phone_home_id
Stale version propagationPhone home matches an older version, run data copied forward

stack-run orchestrator

Type: "stack-run"

Fields:

  • InstallStackID — required
  • InstallStackVersionID — optional
  • RunID — required (the run just created)
  • RequestType — "Create" or "Update"

Execute:

  1. Snapshot roles + inputs (before)
  2. Process outputs (decode, persist, update roles, runner, region, inputs, state gen)
  3. First run? → callback + runner signal + Active status
  4. Newer version exists? → copy run + callback + Active status with provenance
  5. Diff roles → fan out role-change per changed role
  6. Diff inputs → fan out inputs-updated if any keys changed

role-change notification

Type: "role-change"

Fields:

  • InstallID, RoleName, RoleType
  • ChangeType — "enabled" or "disabled"
  • RoleID — ARN / SA email

Execute: No-op (lifecycle hooks deliver notifications)

inputs-updated notification

Type: "inputs-updated"

Fields:

  • InstallID
  • ChangedKeys, AddedKeys, RemovedKeys

Execute: No-op (lifecycle hooks deliver notifications)

await-install-stack-version-run callback

Type: "await-install-stack-version-run"

Changed: Replaced polling with callback-based await

  • Writes CallbackRef to version row
  • Blocks on callback.AwaitWithTimeout
  • Unblocked by stack-run on first run or stale version propagation
